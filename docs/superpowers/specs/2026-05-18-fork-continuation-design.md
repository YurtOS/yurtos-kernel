## Goal

A guest `fork()` returns **twice** — child gets `0`, parent gets the child pid — with the **child resuming at the `fork()` call site holding the parent's exact linear memory and execution state at that point** (true POSIX fork, not a fresh image). **Non-negotiable:** the TypeScript kernel being deleted (#170) supports `fork()`; deleting it without real `fork()` on the Rust/WASM side is a functional regression. Real `fork()` therefore **blocks Phase 4**.

## The two-libc / opt-in-asyncify model (mechanism source-verified)

`fork()` (and `setjmp`/`longjmp`) require capturing and resuming wasm execution state. The only mechanism available on the user-process path is **Binaryen asyncify** (the user-process wasmtime engine has no `async_support`/JSPI/stack-switching — Task-0 spike). Asyncify is a **whole-module control-flow rewrite that taints the module's execution strategy**:

> `abi/toolchain/yurt-toolchain/src/wasm_opt.rs` (≈`:24-26`) — *"Continuations are an explicit opt-in because they taint the process execution strategy: those modules run under the Asyncify adapter while normal modules remain free to use JSPI or another backend."* (text exact; line numbers approximate)

So a module is **either** asyncify-instrumented (runs under the Asyncify adapter; `fork`/`setjmp`/`longjmp` work) **or** lean (free for JSPI/native). It cannot be both — verified, per-module, chosen at build time. Hence **two libcs**:

- **continuation libc** — asyncify-tainted; links the *strong* `fork`/`vfork` shim (`abi/src/yurt_fork.c`) that imports `yurt.host_fork`.
- **lean libc** — no asyncify; preserves the JSPI/native fast route. It links the **always-present weak `fork`/`vfork` ENOSYS stub** (`abi/src/yurt_process.c` ≈`:253-272` — *"wasm32-wasip1 has no fork(); … programs that explicitly call fork …"*) — so `fork()` **compiles and links, and returns `-ENOSYS` at runtime** (it is *not* a link error; the canonical, implemented, test-asserted contract is runtime `-ENOSYS`, per `packages/kernel/src/__tests__/abi_test.ts:620` "keeps plain fork as ENOSYS outside continuation builds"). The continuation archive overrides the weak stub with the **strong shim** in `abi/src/yurt_fork.c` (which imports `yurt.host_fork`).

**Cost of asyncify (why this is opt-in, not universal):**
- Code size — **inherited estimate ≈ +40%**, sourced to the `async-bridge.ts:13-16` comment via `docs/superpowers/plans/2026-05-17-fork-capture-notes.md:150`. **Not re-measured in this worktree** — treat as a prior-knowledge ballpark, not a spike-verified fact.
- A fixed **64 KiB** static unwind side-stack per continuation guest — **directly verified in source**: `abi/src/yurt_setjmp.c:45` `#define YURT_ASYNCIFY_BUF_SIZE 65536` (exported `yurt_asyncify_buf_addr`/`yurt_asyncify_buf_size`).
- Runtime: per-instrumented-function unwind/rewind state-machine prologue/epilogue + local spill, even when never suspending (standard Binaryen-asyncify overhead; not separately measured here).

The two-libc decision rests on the **qualitative** grounds — whole-module asyncify taint (the `wasm_opt.rs` comment above, text-verified) **plus** forfeiting JSPI/native everywhere — and does **not** depend on the exact size percentage. Universal asyncify would impose this on every guest while 99% never `fork()`; opt-in two-libc is therefore correct, not stylistic.

**Open cost lever (recorded, not a blocker):** `continuation_args()` (`wasm_opt.rs:15-18`) applies **blanket `--asyncify`** — no `--pass-arg=asyncify-imports@…` allowlist. Scoping asyncify to only the `fork`/`setjmp` suspend-import call paths would materially cut the +40%/runtime cost. Tracked as a future optimization sub-task; this design does not depend on it.

## The 99% / 1% split (sys_spawn is required, NOT obsoleted)

- **99% — `system()` / `popen()` / `posix_spawn()`**: "run a child program", not "duplicate my address space." Routed through a POSIX extension over **`sys_spawn`** (fork+exec fused, **no continuation, lean libc**). This is the common path; **`sys_spawn` and its contract-completeness (#169) are required and explicitly NOT obsoleted by real `fork()`** — they serve the lean-libc majority. (Corrects an earlier #169 reframing comment that wrongly routed `posix_spawn` through userland fork+exec; a correction is posted to #169.)
- **1% — real `fork()`** (job-control shells, fork-then-work-in-child without exec): continuation libc only. **This spec is the 1% path.**

They are complementary tracks, never to be re-conflated: lean-libc programs get spawn (#169); continuation-libc programs get real fork (this).

## Host architecture

**Current state (Task-0 spike, verified):** the Rust host `host_fork` is a broken **memory-only rebuild** — `snapshot_user_memory` (`kernel_host_interface.rs` ≈`:209`) copies linear-memory bytes only (no stack/locals/return-address); `instantiate_fork_child` (≈`:813`) builds a fresh instance driven via `child.call_run()` (`call_run` def ≈`:3134`) which looks up the `"run"` export, but WASI binaries export `_start`, not `"run"`, so the lookup **errors, the error is swallowed (`Err(_)` → `last_exit()` `None` → `.unwrap_or(127)`), and a phantom child is reaped at exit 127 while `host_fork` returns a real-looking pid**. The JS host `host_fork` is an `-ENOSYS` stub. Both must be **built, not wired** (line numbers approximate).

> **Asyncify is a *guest* mechanism, not host async.** The user-process wasmtime engine has no `async_support`/JSPI/stack-switching — and it does not need it: asyncify is a Binaryen *guest control-flow rewrite* driven through the guest's own `asyncify_start_unwind/stop_unwind/start_rewind/stop_rewind` exports plus the in-linear-memory `yurt_asyncify_buf`. The host only orchestrates those guest exports + the kernel `prepare/commit/rollback`. This is *why* asyncify is the only viable option on the no-host-async path — do not implement it expecting host-side suspension.

> **The TS bridge is the *semantic* oracle, NOT the sequencing template — do not mirror its step order.** Traced in source (`process/loader.ts` fork path + `async-bridge.ts`): the TS path actually `prepareFork` → `snapshotForFork` → **`commitFork` *before the child instance exists*** → `return childPid` synchronously (**parent resumes immediately**) → child instantiate/`restoreForkSnapshot`/resume run **later in a non-awaited async IIFE** (`void childPromise`). It is POSIX-correct only because (a) the parent's full linear memory is deep-copied *inside* `hostFork` at the `fork()` instant, before the parent resumes, and (b) child waitpid-visibility is owned by the *synchronous* kernel `commit_fork` + the `ForkPreparing` filter + the host `-EAGAIN`→block loop, with the deferred child running on the **JS event loop**. **The Rust user-process engine has no event loop and no host async** (stressed above) — it cannot defer the child or block `waitpid` on a not-yet-run child. So Rust must achieve the *same invariants* via a **stricter, serialized order**, explicitly *not* by replicating TS's commit-before-child + non-awaited child (doing so on Rust reintroduces exactly the race the invariant prevents). TS answers *what must hold*; the order below is *how Rust holds it*.

**Rust host** (`packages/runtime-wasmtime`): replace the rebuild with a real asyncify **unwind → snapshot → prepare → restore-child → commit → rewind** dance. The snapshot is NOT taken in the import; it is taken by a **top-level asyncify driver loop** around `_start`/resume (the asyncify protocol the TS `wrapExport` loop demonstrates: `hostFork` only `start_unwind`s + sets a pending flag; the driver, after `stop_unwind`, takes the snapshot — that *mechanic* is shared; the *ordering* below is Rust-specific).

1. **`host_fork` import** (continuation-libc guest): record pending-fork intent, call `asyncify_start_unwind(buf)`, return (value ignored). It does **not** snapshot, prepare, or commit. A **lean-libc** guest reaching `host_fork` (it shouldn't — its weak stub returns `-ENOSYS` before any import) → return **`-ENOSYS`** (fixes the current bug returning a bogus pid). Primary gate is the runtime `-ENOSYS` weak stub, not a link error.
2. **Driver loop** (host-side, wraps `_start`/resume): the guest unwinds frame-by-frame back to the host; loop sees asyncify state == unwinding, calls `asyncify_stop_unwind()`. **Now** (and only now) `yurt_asyncify_buf` + linear memory hold the parent's exact state at the `fork()` site.

(Matches the TS bridge's parent-first/deferred-child shape — *not* a stricter serialized order; rev4's "child runs first" was a wrong over-correction.)

### Execution model — parent-immediate; child a parked *suspendable continuation* drained at the existing `-EAGAIN` point

The Rust user-process engine has **no host async / no event loop**, and the kernel **never truly blocks** — `sys_wait`/`waitpid` without a ready child returns **`-EAGAIN`** (verified `process.rs:1212`; `yurt_abi_methods.toml:86`). The model is **not** "serialized child-first" (rev4, wrong) and **not** TS's event-loop deferral. Parent resumes immediately; the child is parked and cooperatively driven at the `-EAGAIN` re-entry point.

> **Reuse boundary — only the *trigger point* is reused, not the pump's payload (corrects a prior over-claim).** The shipped `drain_and_run_pending_spawns` (#129/#206) is **stateless**: it `instantiate`s a *fresh* `PendingSpawn` child, enters it via `run_start()`/`_start`, and runs it **to completion**. Fork's parked thing is categorically different and **net-new**: (a) a **parked-continuation registry keyed by child pid** — does not exist today (`ProcessForkState` is the 2-variant `Running`/`ForkPreparing{parent_pid}`; it records *that* prepare/commit ran, **no continuation**); (b) the parked entry is a **restored asyncify-instrumented instance mid-rewind**, resumed via `asyncify_start_rewind` into the middle of a call stack — **not** a fresh `_start` (see step 5 / §entry-primitive note); (c) it must be **re-suspendable** — a parked child that itself hits `-EAGAIN` (incl. its own `fork()`) must **re-park, not run to exit**. So the genuinely reused part is *only the parent's `-EAGAIN` drive point*; the registry + re-entrant cooperative drive of suspendable continuations + in-`host_fork` instantiate-and-asyncify-restore are **new and are a primary unknown (see the T2a callout)** — "no new scheduler" / "generalised to fork" was an under-scope; struck.

- After `commit_fork`, the child's restored, **suspendable** asyncify continuation is stored in the new parked-continuation registry keyed by child pid.
- The **parent resumes immediately** and runs until it next hits a kernel call returning `-EAGAIN` (typically `waitpid` on the child). That `-EAGAIN` is the drive point: the host pump runs the parked child until *its* next `-EAGAIN`/exit (re-parking it if not exited), then retries the parent's syscall.
- **Linchpin invariant (makes "kernel unchanged" true — state it, don't derive it):** a *committed-but-not-yet-pumped* child MUST be `waitpid`-visible **as not-yet-exited** — i.e. present in `parent.children`, not reaped, so the parent's `waitpid` returns **`-EAGAIN`** (never `-ECHILD`, never a premature reap) **until its first pump**. This single fact is *why* no new `ProcessForkState` variant and no kernel change are needed; if it ever fails, the "kernel unchanged" premise breaks. T2a must assert it explicitly.
- **Nested/recursive fork is in scope, not hand-waved.** Job-control shells fork pipelines — a parked child forking a grandchild means the pump drive must be **re-entrant** and the parked registry must **nest** (a parked child's own `fork()` parks a grandchild while the child is itself parked). `HOST_WAIT_MAX_DRAIN_ITERS = 100_000` is a *runaway cap, not a nesting-correctness guarantee*. Re-entrant pump + nesting registry correctness is an explicit **T2a** concern (not Out-of-scope; the stated goal — job-control shells — nests).

Supported-pattern boundary (contract):
- **Works:** `fork`→`exec`; `fork`→child works→`exit`, parent `waitpid()`s/reaps; **job-control / background-job shell patterns incl. nested pipeline forks** (parent resumes immediately; children/grandchildren pumped at `-EAGAIN`).
- **Limited (cooperative, not parallel):** the pump advances a child only at the parent's `-EAGAIN` points, so a live parent+child sharing a bidirectional pipe / true coprocess needing *interleaved* progress is cooperative-only; unbounded concurrency stays bounded by `-EAGAIN`/the runaway cap. True parallelism is out of scope — POSIX/job-control fork, the actual goal, is in scope.
- **Named divergence (H3) — detached child whose parent never reaches a drive point.** If the parent `fork()`s a child then does work and **`exit`s without ever `wait`ing** (no `-EAGAIN` drive point ever hit), the child **never runs at all** under the Rust cooperative model — whereas under TS's `void childPromise` event-loop the child eventually runs (and POSIX reparents it to init). `busybox-conformance:110` does **not** exercise this (it only asserts the child does *not* run *before* the parent resumes). This is a real silent cross-host (and POSIX) divergence. **T4 must add a fixture** — parent forks, does work, `exit`s **without `wait`**, child has an observable side effect (writes a file / prints) — to *either* force TS/Rust parity (e.g. drain parked children at process-exit) *or* explicitly ratify this as a documented out-of-scope divergence. It must not stay unnamed; this is exactly the silent gap the parity discipline exists to catch.

## Error handling / edges

- lean-libc `fork()` → **runtime `-ENOSYS`** via the always-linked weak stub (`abi/src/yurt_process.c` ≈`:253+`); single canonical contract, asserted by `abi_test.ts:620`. (Not a link error — see the two-libc section.) `host_fork` reached by any lean guest → also `-ENOSYS`.
- `fork()` from a process with imported shared memory, or a non-main pthread → **`-EAGAIN`**. These guards **already exist** (`snapshot_user_memory` returns `-EAGAIN` on shared memory; `prepare_fork` returns `-EAGAIN` on `threads.len() > 1`) — T2a must **preserve, not regress** them; it is not new first-pass work.
- Any host capture/instantiate/commit failure → `rollback_fork`, no partial child visible to `waitpid`.
- `vfork()` = `fork()` alias (existing non-goal: no suspended-parent shared-address-space semantics).
- `WIFSIGNALED`/full signal wait-status is **#99**, out of scope.

## Testing / cross-host parity

- The Task-0 `fork-twice` characterizing test (`fixture_parity.rs`) is the **tripwire**: today it asserts the rebuild signature (parent-only line, child never runs). On completion it must flip to the true-snapshot signature — **two** lines: the child observing the parent's *pre-fork* sentinel (proving shared memory state at the fork point) with `rc=0`, and parent `rc=child_pid`.
- **Output ordering must be made deterministic, not assumed.** Post-`fork()` parent/child stdout interleaving is not naturally stable. The fixture makes it deterministic via the parked-pump model itself: parent `fork()`s (returns immediately, child parked) → parent **`waitpid()`s → `-EAGAIN` → host pump runs the parked child to exit** → parent reaps → parent prints. Observable byte stream: child line, then parent line, on both hosts (this is the spawn/wait drive applied to fork — consistent with the Execution model, not a coincidental serialization). The cross-host oracle additionally parses the two lines **order-insensitively** (match each by role, assert exact fields: child `rc=0` + pre-fork sentinel; parent `rc=child_pid`) so a future non-`waitpid` fixture can't silently bake in pump order.
- Cross-host **parity oracle**: same `fork-twice` (+ a `fork-exec` fixture) — identical per-field assertions + exit through the Rust `kernel_host_interface` host AND the JS `Runner`, same discipline as the merged spawn/wait slice (#129).
- **The existing Rust-lifecycle fork oracle is CI-dark — do not lean on it (M1, mechanism corrected).** `sandbox-wasm-kernel_test.ts:179` ("runs fork continuations through Rust-owned process lifecycle") is **referenced by no workflow at all** — it is *not* in `deno.yml`'s curated `IMAGE_RUNTIME_TESTS`, so it **never runs in CI**, independent of its `!HAS_JSPI || !hasKernelWasm()` early-return (don't assume re-enabling JSPI/kernel-wasm CI would resurrect it — it wouldn't). The fork cases that run green on every PR (`abi_test.ts:620/:640` via `guest-compat.yml`) exercise the **TS/JSPI** path, not Rust. **T4 must add a new, unconditionally-running Rust-host fork parity test** wired into a workflow's actual run list — leaning on the existing dark test would pass vacuously.
- Fork fixtures must be built through the **continuation/asyncify toolchain mode** (`wasm_opt.rs` `use_continuation=true`). A working asyncify build path **already exists** in the repo (`abi/Makefile` ≈`:201-207` + `wasm_opt.rs:17`; it builds the existing `fork-canary.wasm` the current TS oracles consume). The gap is narrow: the **Rust fixture harness** `ensure_fixture_built` (`fixture_parity.rs`) does plain `cargo build` with no `wasm-opt --asyncify` step — T1 wires it to the *existing* asyncify path, it does not invent one.

## Decomposition (for the implementation plan)

1. **T1 — CRITICAL PATH / longest pole.** `fork-twice` (exists) + `fork-exec` fixtures **and the asyncify fixture-build harness** (`ensure_fixture_built` has no `wasm-opt --asyncify` path; `wasm_opt.rs use_continuation` mode must be wired in). **(M2)** T1 also delivers a **minimal unconditionally-CI'd Rust-host fork oracle** (initially red — characterizes the broken rebuild like the Task-0 `fork-twice` test, then becomes the T2a green target) so **T2a iterates against a live oracle from day one** instead of being validated only at T4. This is the least-specified, highest-unknown item and **gates both hosts**. Plan it first, with the most buffer, and de-risk it before committing T2a/T3 estimates.
2. **T1.5 — Rust-host setjmp/longjmp substrate (H1 resolution; prerequisite of T2b ONLY).** Port the proven TS `async-bridge.ts` machinery to the Rust host: `host_setjmp`/`host_longjmp` imports, `pendingSetjmp`/`pendingLongjmp`, the `jmpBufStates` map, `captureBuffer`/`restoreBuffer`, snapshot/restore. **Bounded** — a port of a working reference, not open-ended design (per H1: the Rust host has *zero* such surface today; setjmp/longjmp state is host-owned, so this map cannot be skipped for longjmp-across-fork). **Independently valuable:** this is exactly the missing impl the pending-on-impl legacy Deno setjmp/longjmp tests wait on ([setjmp/longjmp = libc + asyncify]) — re-run and re-gate them when it lands. **Does NOT gate T2a or basic fork** (empty `jmpBufStates` ⇒ trivial); gates **T2b** only. Logically independent of T1; can run in parallel with T1/T2a/T3.
3. **T2a — fork core (co-equal high-unknown with T1/T3; its hard core was previously under-scoped).** Rust host: replace the rebuild with real asyncify snapshot/rewind; lean-guest `host_fork` → `-ENOSYS`. **The downplayed core (Reuse-boundary callout in Execution model):** the parked-continuation **registry**, the **re-entrant cooperative drive of *suspendable* (re-parkable) continuations** incl. **nested** parked children (job-control pipelines), the synchronous in-`host_fork` instantiate-and-asyncify-restore, and the linchpin `waitpid`-visible-as-not-exited invariant — are **all net-new**; only the parent's `-EAGAIN` trigger point reuses `#129/#206`. Do **not** copy the spawn pump's stateless `run_start()`/run-to-completion payload. **No setjmp dependency** (empty `jmpBufStates`); green against `fork-twice`/`fork-exec`/T1's oracle/`abi_test.ts:620`. Plan with buffer comparable to T1; *not* a cheap parallel leg.
4. **T2b — `jmpBufStates` capture+restore across fork (depends on T1.5 + T2a).** Extend the T2a snapshot to the 4th part: capture the parent's `jmpBufStates` at the `fork()` site and restore it into the child so pre-fork `setjmp`/`longjmp` frames survive. Oracle = the `abi_test.ts:640` longjmp-across-fork canary (behavioral). Small relative to T2a *given* T1.5; meaningless without it.
5. **T3** — JS host: port-and-adapt the `AsyncifyAsyncBridge` **and its kernel integration surface** (see the Host-architecture T3 callout — larger than a symbol copy). Logically independent of T2a/T2b's code, but **both are heavy** — see critical path.
6. **T4** — cross-host parity (`fixture_parity.rs` + JS `Runner` E2E) + edges (`vfork`, `-EAGAIN` shared-mem/threaded, `-ENOSYS` lean). **Sub-item (mandatory):** add a **new unconditionally-CI'd Rust-host fork oracle** — the existing Rust-lifecycle test is CI-dark (see Testing); without this T4 passes vacuously. **Depends on T1 + T2a + T3** for the core parity; the longjmp-across-fork sub-oracle (`abi_test.ts:640` Rust equivalent) additionally **depends on T1.5 + T2b**.
7. **T5** — verify + annotate #170 with the precise *keep-until-T4* file list: `async-bridge.ts`; `process/loader.ts`; `process/module-profile.ts` (its `importsFork→requiresAsyncify` rule); the oracles `abi_test.ts:620/:640`, `sandbox-wasm-kernel_test.ts:179`, `busybox-conformance_integration_test.ts:110`. **Not** `manager.ts`. Confirm the file-scoped interlock, not a blanket #170 block.

**Critical-path shape:** `T1 → (T2a ∥ T3 ∥ T1.5) → T2b → T4 → T5` *structurally* (T2b needs T2a + T1.5; T4 core needs T2a + T3, its longjmp-across-fork sub-oracle needs T2b), but **schedule-risk-wise treat T1, T2a, and T3 as three co-equal high-unknown poles, not "T1 hard, the rest cheap."** T2a's parked-suspendable-continuation core (above) rivals T1 and T3 in unknowns; the earlier "reuses the proven pump / T2 independent-and-easy" framing under-stated it and is corrected. **T1.5 is *bounded* (a port of the proven TS `async-bridge.ts` machinery), not a high-unknown pole** — but it is on the path to T2b and to retiring the legacy setjmp/longjmp tests, so it must be scheduled, not deferred. T2b is small *given* T1.5. De-risk T1/T2a/T3 with spikes before committing dates; the plan should sequence/parallelize on real spike outcomes, not assume any of the three is the cheap leg.
