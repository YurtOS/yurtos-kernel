# B2.9 — `lstat()` no-follow stat across ABI + host trampolines (issue #81)

Issue #81, split off from #67 (PR #80). Part of the full-parity initiative
(tracking #52 / #71, umbrella #57). Developed stacked on #67's branch (which
made `stat()` follow symlinks and deliberately preserved the no-follow lexical
path — exactly what `lstat` needs). **#80 has since merged to `main`** (squash
commit `d0c747c`); this slice has been **rebased onto `origin/main`** and is a
standalone B2.9 slice off `main` (no longer stacked). The single lstat commit
replants cleanly atop main's #80 + #72 + #101 (`take_bytes`) — verified by the
full gate post-rebase.

## Problem (issue #81)

`lstat` is absent end-to-end: no `[method.sys_lstat]` in
`abi/contract/yurt_abi_methods.toml`, no `lstat_path` / `METHOD_SYS_LSTAT` in
`packages/kernel-wasm/src/`, and no guest trampoline in any of the three host
adapters. After #67, `stat()` follows symlinks (POSIX-correct); a program that
needs to see the *link itself* (`test -L`, `find -type l`, `cp -P`, `ls -l`,
`os.path.islink`, `fs::symlink_metadata`) has no syscall to call.

## Architecture note — `host_lstat` is a 1:1 guest trampoline, **not** a real-FS call

Two distinct host-side surfaces, often conflated:

- **`host_lstat` / `sys_lstat` (this slice).** The guest libc trampoline. Guest
  `lstat()` → libc emits the `host_lstat` (JS/Deno) / `sys_lstat` (wasmtime)
  import → the adapter forwards it **1:1 to the kernel's `SYS_LSTAT` dispatch**.
  The **kernel VFS shim answers**. Needed in all three adapters or a guest
  cannot call `lstat()` at all.
- **`kh_real_stat` / host-disk `symlink_metadata` (NOT this slice).** Real
  host-FS passthrough for host-mounted paths. We deliberately do *not* route
  `stat`/`lstat` to the host FS: the sandbox VFS view is the source of truth and
  intentionally differs from the host (uid/gid, modes, namespace). A separate
  concern, out of scope here.

So the adapter change is pure forwarding glue mirroring `host_stat` exactly — no
new backend logic, no `HostFsImpl` method.

## Scope (B2.9 — cargo + deno unit-testable)

ABI: `[method.sys_lstat] id = 0x1_0078`, `kind = "syscall"`. `0x1_0078` is the
next free id in the **B2 fd/vfs block `0x1_0070–0x1_007F`** (PR #55; used through
`0x1_0077` `sys_ioctl`), collision-free per the method-id partition rule. The
kernel-wasm `build.rs` generates `METHOD_SYS_LSTAT` from this entry.

Kernel (`packages/kernel-wasm/src/dispatch/`):

- Extract the no-follow typing tail of `stat_path` into a shared
  `write_stat_record(k, path, response)` helper (unix-socket inode → `S_IFCHR`;
  `entry_type` → `S_IFDIR`/`S_IFLNK`/`S_IFREG`; size+mode; 16-byte record;
  `-ENOENT` for filetype 0). `stat_path` = `normalize` → `follow_symlinks` →
  `write_stat_record`. `lstat_path` = `normalize` → `write_stat_record` (the
  follow step is the *only* difference, so the two cannot drift).
- `mod.rs`: import `lstat_path`; add `METHOD_SYS_LSTAT => lstat_path(...)` arm.

Because `normalize_readable_path` is lexical (no-follow) and `entry_type` does
not resolve links, `lstat` on a symlink (to dir, to file, **or dangling**)
reports `S_IFLNK` and returns 16 — it never `-ENOENT`s on a live link, the
mirror image of #67's `stat` tests.

Scope note (pre-existing, not introduced here): POSIX resolves symlinks in
every *intermediate* path component and only the *terminal* component differs
between `stat` (follow) and `lstat` (no-follow). `normalize_readable_path` is
fully lexical, so intermediate symlink components are resolved for **neither**
`stat` nor `lstat` — a limitation inherited from #67's `stat`, identical and
consistent across the two. Faithful component-wise resolution is a VFS-wide
concern (cf. the #59 dirfd inode-anchoring track), out of scope for this slice.

Host trampolines (1:1 forward, mirror `host_stat`):

- JS `kernel-host-interface-js/mod.ts`: `SYS_LSTAT: 0x1_0078` in the `METHOD`
  table (consumed by Deno `HOST_BINDINGS` and the wasmtime drift test's source
  of truth — the TOML).
- Deno `kernel-host-interface-deno/wasm-kernel-imports.ts`: `HOST_BINDINGS`
  entry `{ name: "host_lstat", method: METHOD.SYS_LSTAT, args: ["ptr_len",
  "out_cap"], returnsBytes: true }` immediately after `host_stat` (identical
  wrapper shape — 16-byte record).
- wasmtime `runtime-wasmtime/src/kernel_host_interface.rs`: `mod sys_method_id`
  `pub const LSTAT: u32 = 0x1_0078;`; a `linker.func_wrap(SYS_NAMESPACE,
  "sys_lstat", …)` block mirroring `sys_stat` (`sys_method_id::LSTAT`, out 16).

## Testing (TDD)

Kernel `#[cfg(test)]` via `dispatch()`, red→green, mirroring the #67 stat tests
but asserting filetype `7` (`S_IFLNK`):

1. `lstat` on symlink→dir → filetype 7 (NOT 3).
2. `lstat` on symlink→regular → filetype 7, size 0 (the link's, not target's).
3. `lstat` on dangling symlink → returns 16, filetype 7 (NOT `-ENOENT`; the
   link exists).
4. `lstat` on a regular file → identical bytes to `stat` (size + mode).
5. `lstat` on a directory → filetype 3 (matches `stat`; no link to differ on).
6. `lstat` on a missing path → `-ENOENT`.

Refactor discipline: the existing #67 `stat_*` tests guard `write_stat_record`
extraction (must stay green through the refactor).

Parity gates (the issue's "parity green"):

- Deno contract test `wasm-kernel-imports_test.ts` — add `host_lstat` to the
  binding-coverage assertions + a wrapper exercise (symlink then `host_lstat`
  reports `S_IFLNK`), mirroring the `host_symlink`/`host_readlink` test.
- wasmtime drift test `kernel_host_interface_method_ids_match_yurt_abi_methods_toml`
  — add `("sys_lstat", METHOD_SYS_LSTAT, …)` tuple + test-local

## Residual (tracked, not a bug)

POSIX `lstat` reports `st_size` = the length of the symlink's target
string. `write_stat_record` returns size `0` for filetype 7 (and for
all non-regular types), matching the sandbox VFS's existing fixed-size
convention for non-regular entries (dirs, sockets) — it is *intended*
and documented in the `sys_lstat` ABI doc, not a defect of this slice.
The only observable gap: a guest that pre-sizes a `readlink` buffer
from `lstat().st_size` (instead of growing on truncation) would
under-size it. No in-tree guest does this. Reporting the real link
length would touch the shared `write_stat_record` filetype-7 arm only
(`stat_path` follows links, so it never types an entry as 7) — a clean
follow-up if a real guest needs it, deliberately **not** scoped here to
keep this slice additive and the helper drift-free. Flagged by PR #120
review; mirrors the "tracked residual" discipline of the B2.8 / #60
spec.
