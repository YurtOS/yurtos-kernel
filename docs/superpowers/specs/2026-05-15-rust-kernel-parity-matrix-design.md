# Rust Kernel Parity Matrix

This is the retirement checklist for replacing the TypeScript kernel authority
with `packages/kernel-wasm` plus host adapters. `Status` is deliberately about
retirement readiness, not whether a first implementation exists.

This document is also the **canonical living tracker** for the full-parity
initiative (umbrella PR `rust-kernel-full-parity`, GitHub tracking issue mirrors
it). It is updated by every slice PR.

## Scope changes & superseded non-goals (2026-05-16)

The parity definition for this initiative is **full Linux-userland parity
(maximal)**: the ~80–120 real-userland syscall target (master checklist:
`reference_linux_syscall_table.md` → filippo.io/linux-syscall-table). This
**supersedes** narrower non-goals in earlier shipped specs. Those specs are not
edited retroactively; this section is the authoritative override:

- `2026-05-04-posix-libc-runtime-hardening-design.md` — "No fork/vfork
  implementation": superseded. `fork()` is implemented; **true `vfork`
  address-space sharing is now in scope** (slice B1).
- `2026-05-16-rust-fork-parity-design.md` non-goals (multithreaded fork,
  `pthread_atfork`): **now in scope** as tracked B1 sub-slices.
- `host_dns_resolve` was "deferred extension": **now in scope** (slice B3). The
  relevant `project_*` memory note must be revised when B3 lands.
- Previously-deferred maximal items now in scope and tracked as rows below:
  AF_INET6, RT signal queueing, `pthread_cancel`, S3 `VfsBackend`.

## Sequencing (approach C) & definition of done

Thin gate first, then slices, then harden gate, then TS retirement:

- **B0 — thin parity gate** (infra, not a syscall row): add a
  `YURT_KERNEL=ts|wasm|both` runtime switch and a TS-vs-Rust differ over
  `abi/conformance/*.spec.toml` +
  `packages/runtime-wasmtime/tests/fixture_parity.rs`, wired into the slow tier
  (`guest-compat.yml`). Also vendors + wires the Open POSIX Test Suite (see
  below) with a smoke subset green. Unblocks objective measurement for every
  later slice.
- **B1** process model completion · **B2** FD/VFS · **B3** sockets/network ·
  **B4** persistence/extension · **B5** harden gate (every row CI-locked +
  **full Open POSIX Test Suite green**) · **B6** TS retirement (reduce
  `packages/kernel/*` to adapters, default `YURT_KERNEL=wasm`).

A row is `done` only when: Rust owns the state, all KH adapters are wired, the
named fixtures pass, the B0 differ shows zero TS-vs-Rust behavioral diff,
**and** the row's Open POSIX Test Suite interface area is PASS (or every
non-PASS is a justified tracked `UNSUPPORTED` exception). `Verified@` records
the commit at which that last held.

### Method-id partition by slice (canonical)

Each sub-goal is its own PR off `main`, so they append new `[method.sys_*]` ids
concurrently. To honor the contract's never-reuse/never-renumber rule regardless
of merge order, the `0x1_00xx` syscall id space is partitioned into per-slice
blocks (mirrored in `abi/contract/yurt_abi_methods.toml`):

| Slice              | PR  | Method-id block     | Used so far           |
| ------------------ | --- | ------------------- | --------------------- |
| B1 process model   | #54 | `0x1_0060–0x1_006F` | `0x1_0060`…`0x1_0065` |
| B2 fd/vfs          | #55 | `0x1_0070–0x1_007F` | `0x1_0070`,`0x1_0071` |
| B3 sockets         | —   | `0x1_0080–0x1_008F` | —                     |
| B4 persistence/ext | —   | `0x1_0090–0x1_009F` | —                     |

Future slices take the next free block. New ids append within the owning slice's
block only.

### POSIX conformance corpus (hard gate)

The IEEE 1003.1-2001 **Open POSIX Test Suite** (posixtest.sourceforge.net) is
adopted as an external conformance corpus. Vendored under
`abi/conformance/posix/`:

- **Primary:** `github.com/bytecodealliance/open-posix-test-suite` — the
  Bytecode Alliance WASI/`wasi-libc` fork; matches our `wasm32-wasip1` guest
  model directly.
- **Secondary cross-check:** `github.com/emscripten-core/posixtestsuite`.

Scope = the `conformance/interfaces/` tests for in-scope areas (pthread,
signals, semaphores, clocks, timers, sched, mqueue, …). Each test is built to
wasm, run through the B0 differ under both kernels, and graded by the standard
suite result codes: **PASS=0, FAIL=1, UNRESOLVED=2, UNSUPPORTED=4, UNTESTED=5,
HUNG=6**. "Pass them all" = every applicable test PASS under the Rust kernel
**and** identical TS-vs-Rust; any non-PASS must be an explicit, written-reason
`UNSUPPORTED`/deferred matrix exception (physically-wasm-impossible cases only)
— **no silent skips**. B0 lands the harness + a smoke subset; full-corpus green
is the B5 gate, tracked by a published per-area `posix-suite` scoreboard that CI
enforces as non-regressing.

## Coverage expansion (living)

The rows above are the retirement-authority rows. Full-userland coverage
(~80–120 syscalls) is expanded **per slice**: each slice spec enumerates the
exact syscall list it owns and adds/splits rows here as it lands. Speculative
pre-enumeration of all 467 Linux entries is intentionally avoided — the matrix
grows with verified work, not ahead of it. The master checklist for "what
userland actually needs" is `reference_linux_syscall_table.md`.

## Deferred Compatibility Imports

There are no known transitional pthread `host_*` imports left that require the
TypeScript kernel to own lifecycle state.

`host_dns_resolve` is **no longer deferred** — it is in scope for slice B3 (see
"Scope changes" above). When B3 lands, revise the corresponding `project_*`
memory note that currently treats DNS as off-path.

Any future `[method.sys_*]` entry must either have a `METHOD_SYS_*` dispatch arm
in `packages/kernel-wasm/src/dispatch/` or appear above with a non-`done`
`Status` and an owning `Slice`.
