# B2.9 — openat/fchdir inode-anchoring (VfsBackend dir-handle API)

Sub-goal slice of the full-parity initiative (tracking #52, umbrella #57).
Converts issue **#59** into a planned PR. Off `main`; no new syscalls / ABI
method-ids. **This is a substantial VFS rewrite, not a dispatch-layer patch** —
issue #59's "its own session that may rewrite the VFS" framing was correct; an
earlier draft of this spec under-scoped it (see Revision note). Spec corrected
per PR #63 review before plan/TDD.

## Problem

`openat(dirfd, path, flags)` and `fchdir(dirfd)` are not POSIX-faithful: a
directory fd must track the directory **object** (inode), so resolution stays
correct after the directory is renamed/unlinked. Today
`FdEntry::Directory { path: Vec<u8> }` (`kernel.rs:153`) stores only the path
**snapshot** captured at `open()`; `sys_openat` (`dispatch/fs.rs:178`, the
`FIXME(#59)` site) clones it and joins the child, so after a rename it resolves
the stale path. `fchdir`/`chdir`/`getcwd` share the root cause
(`Process.cwd: Vec<u8>`, `kernel.rs:300`).

```
fd = open("/base", O_DIRECTORY)
rename("/base", "/renamed")
openat(fd, "x", O_CREAT|O_WRITE)   // POSIX: /renamed/x ; actual: /base/x (wrong)
```

### Why this is bigger than "expose inode→path" (PR #63 review)

The earlier draft claimed "every backend already allocates stable inode ids and
keeps a path→inode map [for directories]". **That is false for the primary
backend.** `RamfsBackend` (`vfs.rs:450`) tracks directories as
`dirs: BTreeSet<Vec<u8>>` — _pure path strings, no inodes_.
`paths: BTreeMap<Vec<u8>, u64>` is **files-only**. There is no directory inode
to anchor a dirfd to, so it must be _introduced_.

Worse, the whole ramfs namespace is **path-string-keyed, flat**: a file lives at
key `/base/x` in `paths`. `RamfsBackend::rename` of a directory (`vfs.rs:724`,
dir branch ≈ `vfs.rs:760`) does only
`self.dirs.remove(old); self.dirs.insert(new)` — it does **not** rewrite child
keys. After `rename("/base","/renamed")`, `/renamed` is an empty dir with
orphaned children still keyed `/base/x`. So even a perfectly stable `dir_inode`
resolves to nothing — **recursive subtree re-key on directory rename is an
unstated hard prerequisite** of the headline DoD example, and it _is_ new
behavior.

## Approach (trait dir-handle API + ramfs inode-tree + recursive rename)

Three pieces, sequenced (Sequencing section). The trait shape (Approach A,
already approved) is kept; the corrections add the missing prerequisites.

### 1. `VfsBackend` trait additions (default-`None`, opt-in)

```rust
trait VfsBackend {
    /// Stable inode of `path` iff it is a directory. None ⇒ backend
    /// does not support inode-anchored dirs (caller uses the
    /// path-snapshot fallback for that mount — see Degraded mode).
    fn dir_inode(&self, path: &[u8]) -> Option<u64> { None }
    /// Resolve ONE path component `name` directly under `dir_inode`.
    /// Returns (child_inode, filetype) where `filetype` is the
    /// EXISTING WASI filetype byte returned by `entry_type` (trait
    /// default `vfs.rs:179`; `RamfsBackend` impl `vfs.rs:712`:
    /// 0=UNKNOWN, 3=DIRECTORY, 4=REGULAR_FILE, 7=SYMBOLIC_LINK) —
    /// NOT a new parallel encoding.
    ///
    /// `child_inode` is required only when the caller can continue an
    /// inode-anchored walk through the child (directories; regular
    /// files if the backend already has file inodes). Symlinks may
    /// return `(None, 7)`: dispatch never follows symlinks through
    /// `resolve_at`; it reconstructs the absolute path and delegates to
    /// the existing path-based resolver. This avoids requiring ramfs to
    /// invent symlink inodes or fake inode values; ramfs symlinks remain
    /// path->target entries in this slice.
    fn resolve_at(&self, dir_inode: u64, name: &[u8]) -> Option<(Option<u64>, u8)> { None }
    /// Reverse: current absolute (mount-relative) path of a dir
    /// inode, or None if the inode is no longer a live directory
    /// (rmdir/unlinked) — distinct from "backend doesn't support it".
    fn dir_path(&self, dir_inode: u64) -> Option<Vec<u8>> { None }
}
```

`MountTable` gains thin pass-throughs returning `(MountId, inode)` /
`(MountId, Option<inode>, filetype)`, mirroring the file-OFD `(mount_id, inode)`
pattern (`OpenFileDescription`, `kernel.rs:488`) while preserving the "symlink
has no inode" case above.

### 2. Ramfs directory-inode model + recursive subtree rename (HIGH effort)

`RamfsBackend` directories become inode-bearing:

- Replace `dirs: BTreeSet<Vec<u8>>` with a structure that mints a stable inode
  per directory from `RamfsBackend`'s existing per-backend `next_id` counter
  (`vfs.rs:469`) — the same counter `install` already uses for file inodes, so
  dir and file inodes share one collision-free id space _within ramfs_. (The
  "single global inode-id space" note at `vfs.rs:23` is a Phase-3 design-intent
  doc comment, **not** one shared counter — each backend allocates from its own
  `next_id`: ramfs `vfs.rs:469`, proc `vfs.rs:1221`, overlay `vfs.rs:1835`.)
  Concretely: a `dir_inodes: BTreeMap<Vec<u8>, u64>` plus the reverse
  `dir_paths: BTreeMap<u64, Vec<u8>>`. `mkdir` mints; `rmdir` frees both sides;
  `/` gets a fixed root dir-inode at construction.
- `RamfsBackend::rename` of a directory must **recursively re-key** every
  descendant in `paths`, `symlinks`, `dir_inodes`/`dir_paths` from the old
  prefix to the new prefix (inode ids are preserved — only the path keys move),
  so children remain reachable and open file OFDs (which hold `inode`, not path)
  are unaffected. This is the prerequisite that makes the DoD example pass.
- `dir_inode(path)` = `dir_inodes.get(path)`; `dir_path(inode)` =
  `dir_paths.get(inode)`; `resolve_at(dir_inode, name)` = look up
  `dir_paths[dir_inode]`, join `name`, classify via the existing `entry_type`,
  return `(Some(child_inode), filetype)` for directories and regular files (from
  `dir_inodes`/`paths`), and `(None, 7)` for symlinks (from `symlinks`) so the
  dispatch takes the symlink fallback.
- **Isolation invariant:** dir inodes share the `next_id` space but must **not**
  enter `inodes` (content buffers, `vfs.rs:451`) or `refcount` (hardlink counts,
  `vfs.rs:468`) — directories have no byte content and no hardlinks. `rmdir`
  frees `dir_inodes`/`dir_paths` only. The recursive re-key moves _path keys_
  while preserving every file inode id and its `refcount`/`inodes` entry
  untouched, so open file OFDs (which hold `inode`, not path) survive a
  parent-dir rename unchanged — this is what makes the "OFDs unaffected" claim
  above hold.

Read-only/synthetic backends do **not** need rename (immutable), but DO need
`dir_inode`/`resolve_at`/`dir_path` if their dirs are to anchor:

- **proc** (`vfs.rs:1214`): currently `inodes: BTreeMap<Vec<u8>, u64>` is
  populated for the synthetic _files_ it lists; verify it has a dir-inode space
  for `/proc`, `/proc/<pid>` — if not, mint synthetic stable dir inodes the same
  way (still low effort: paths are stable, no rename).
- **tar** (`TarLayerBackend`, `vfs.rs:1604`): immutable; `inodes` covers files.
  Dir inodes are derivable from the path index; low effort, no rename.
- **dev** (`vfs.rs:548`): fixed namespace; hardcode the `/` dir inode +
  `resolve_at` table. Trivial, no rename.
- **overlay** (`vfs.rs:1822`): HIGH — `dir_inode`/`resolve_at`/`dir_path` must
  demultiplex upper/lower, honor whiteouts, and survive copy-up; recursive
  rename composes lower+upper. Treat as its own sub-step; if not completable
  in-slice, overlay mounts use the **degraded mode** below and it is a tracked
  matrix exception (not silent).
- **hostfs** (`vfs.rs:1429`): host renames are invisible to us → permanent
  **degraded mode** (path-snapshot), tracked matrix exception.

### 3. Dispatch rewrite + dual representation + mount boundaries

- `FdEntry::Directory` becomes a dual capability:
  ```rust
  Directory {
      mount_id: MountId,
      // inode-anchored when the owning backend supports dir inodes;
      // None ⇒ path-snapshot fallback for backends that return None
      // from dir_inode (hostfs, overlay-if-deferred, embedder backends).
      dir_inode: Option<u64>,
      path: Vec<u8>, // always kept: the snapshot fallback AND the
                     // cheap getcwd/readdir source when inode-anchored
  }
  ```
  `Process.cwd` becomes the same shape (replacing `cwd: Vec<u8>`). `path` stays
  the `PathResolver`-normalized **absolute** snapshot, so the existing
  `debug_assert!(dir.first() == Some(&b'/'))` invariant (`dispatch/fs.rs:197`)
  is preserved, not removed. `PathResolver::{normalize,realpath}` must not read
  `cwd.path` blindly. Before resolving any relative path, it asks `MountTable`
  for the current `dir_path(cwd.dir_inode)` when `dir_inode` is present,
  composes that mount-relative path with the mount's absolute prefix, and
  refreshes `cwd.path`. If `dir_inode == None`, it uses the cached snapshot
  degraded mode; if `dir_inode` is present but `dir_path` returns `None`, the
  relative resolution fails `ENOENT` rather than using a stale pre-removal path.
  This is a process-wide invariant, not an `openat` special case: relative
  `open`, `mkdir`, `rmdir`, `unlink`, `rename`, `link`, `symlink`, `chmod`,
  `chown`, `utimens`, `stat`, `readlink`, `realpath`, `spawn`/exec path lookup,
  `/proc/<pid>/cwd`, and any future `PathResolver` caller inherit the refreshed
  cwd behavior automatically. **Degraded mode (default-`None` contract):** when
  `dir_inode` is `None`, `openat`/`fchdir`/`getcwd` behave exactly as today
  (path-snapshot) for that fd/mount — correct for unchanged dirs, stale across
  rename, and a _documented, tracked_ limitation for that backend (never a
  silent skip).
- `sys_openat`: if the dirfd is inode-anchored, walk the relative path
  **component-by-component** via `MountTable`-level resolution starting at
  `(mount_id, dir_inode)`:
  - Dot-component rule: before the inode walk calls backend `resolve_at`, scan
    the relative path components. If any component is `.` or `..`, STOP the
    inode walk, reconstruct the absolute path from the current
    `dir_path(dir_inode)` plus the original relative path, compose the mount
    prefix, and re-delegate to the existing path-based `sys_open` /
    `PathResolver` path. This keeps lexical normalization centralized and
    preserves today's `openat(fd,"./child")` and `openat(fd,"../sibling")`
    behavior, including mount-root handling. If `dir_path(dir_inode)` is already
    `None`, return `ENOENT` for lookup/create.
  - **Mount-boundary rule (PR #63 review [P2]):** at each component, if
    advancing would cross into a _child mount_ (the component path matches
    another mount's prefix), STOP the inode walk, reconstruct the absolute path
    (`dir_path(dir_inode)` + remaining components), compose the mount prefix,
    and re-delegate to the existing path-based `sys_open` (which routes through
    `MountTable::resolve` longest-prefix). This preserves
    `openat(rootfd,"proc/1/cmdline")` crossing ramfs→ProcBackend, which the
    current path-join+`sys_open` gets right and a single-backend walk would
    break.
  - Symlink mid-walk or any non-dir component: same reconstruct-absolute + fall
    back to `sys_open` (keeps the 40-hop SYMLOOP logic centralized, not
    duplicated in the walk). A symlink result does **not** require a child
    inode; filetype 7 is enough to trigger this fallback.
  - dirfd not inode-anchored (`dir_inode == None`): current behavior (join
    stored `path` + `sys_open`).
  - Absolute path / `AT_FDCWD`: unchanged (delegate to `sys_open`).
- `fchdir` copies the fd's `(mount_id, dir_inode, path)` into `Process.cwd`
  (inode-anchored when available, else snapshot). `chdir` resolves the new path
  to `dir_inode` via `MountTable` (None ⇒ snapshot, degraded). `getcwd` returns
  the mount-absolute form of `dir_path(cwd.dir_inode)`, refreshing `cwd.path`
  with `mount_prefix(mount_id) + dir_path` (normalizing `/` so a cwd at a
  backend mounted on `/mnt` reports `/mnt`, not `/`). If `dir_inode` is `None`
  _or_ `dir_path` returns `None` (dir was removed) it returns the last cached
  `cwd.path` (POSIX-ish "(unreachable)" is out of scope).

## Testing (TDD red→green)

1. **rename-stability (the DoD):** `fd=open("/base",O_DIRECTORY)`;
   `rename("/base","/renamed")`; `openat(fd,"x",O_CREAT|O_WRITE)` creates
   **`/renamed/x`**; reading `/renamed/x` returns the bytes. (Requires the
   recursive-rename prerequisite — see the dedicated rename test below.)
2. **recursive subtree rename (prerequisite, its own test):** create
   `/base/sub/f`; `rename("/base","/renamed")`; assert `/renamed/sub/f` is
   readable and `/base/sub/f` is gone — _independent of dirfds_, locking the
   ramfs re-key.
3. **removed-dir create → ENOENT (PR #63 review [P2]):**
   `fd=open("/d",
   O_DIRECTORY)`; `rmdir("/d")` (or unlink of the dir); then
   `openat(fd,"child",O_CREAT)` returns **`-ENOENT`** (a removed directory has
   no linkable path — POSIX), explicitly _distinct_ from the rename case in
   test 1. `dir_path(dir_inode)` returns `None` once removed and the dispatch
   maps that to ENOENT for create/lookup.
4. **mount-boundary:** with `/proc` mounted (ProcBackend),
   `fd=open("/",O_DIRECTORY)`; `openat(fd,"proc/self/cmdline",...)` resolves
   through the mount (re-delegates), not "miss inside ramfs".
5. **fchdir/getcwd across rename:** `fchdir(fd_of_/base)`; `rename`; relative
   `open("y",O_CREAT)` lands under `/renamed`; `getcwd` → new path. Include the
   same assertion under a non-root mount so a backend root mounted at `/mnt`
   reports `/mnt` and `/mnt/child`, never mount-relative `/` or `/child`.
6. **degraded mode:** a backend returning `None` from `dir_inode` (e.g. a hostfs
   mount, or a test stub) → `openat`/`fchdir` behave as the path-snapshot
   (asserted, so the fallback is _defined behavior_, not an accident).
7. **symlink mid-walk fallback:** `symlink("/target","/base/sym")` where
   `/target` is a dir containing `f`; `fd=open("/base",O_DIRECTORY)`;
   `openat(fd,"sym/f",…)` resolves `/target/f`. Asserts the component walk sees
   filetype 7, reconstructs the absolute path, and re-delegates to `sys_open`
   (centralized 40-hop SYMLOOP), i.e. it does **not** treat a symlink component
   as a directory.
8. **fork inherits cwd inode:** child inherits / `fchdir`s an inode-anchored
   `cwd`; the **parent** then renames that dir; the child's subsequent
   _relative_ `open` resolves under the new path. Locks `Process.cwd`
   inode-inheritance across `clone`/`fork` (the `cwd: p.cwd.clone()` site,
   `kernel.rs:1299`) — distinct from test 5, which renames within a single
   process.
9. **per-backend `resolve_at`/`dir_inode`/`dir_path` unit tests** for ramfs,
   proc, tar, dev (+ overlay if in-slice).
10. **dot-component inode-dirfd fallback:** after opening `/base` as a directory
    fd, assert `openat(fd,"./child",...)` resolves exactly like
    `openat(fd,"child",...)`; after rename, assert
    `openat(fd,"../renamed/child",...)` uses the refreshed absolute base and
    centralized `PathResolver` normalization, not literal backend lookups for
    `.`/`..`.
11. **all relative-path cwd refresh:** `fchdir(fd_of_/base)`; rename `/base` to
    `/renamed`; then relative `mkdir`, `unlink`, `chmod`, `realpath`, and
    exec/spawn lookup use `/renamed` via the refreshed cwd, not the cached
    pre-rename snapshot. If the inode-backed cwd has been removed and `dir_path`
    returns `None`, the same relative operations fail `ENOENT` instead of using
    the stale cached path.
12. **conformance canary** for rename-stability, locked through the B0 differ
    (TS-vs-Rust). If TS diverges (likely — TS dirfd is also a path snapshot),
    the differ emits a candidate `parity-baseline.toml` `[[divergence]]` row
    shipped in this same PR; per that file's documented process the reviewer
    assigns the owning slice (`slice = "B<n>"`, B1..B6 — this reads as a
    TS-retirement divergence) with reason "Rust POSIX-faithful dirfd; TS
    path-snapshot — retired with the TS kernel", per the no-silent-skip
    discipline.

## Non-goals

- No new syscalls / ABI method-ids; `host_*` surface unchanged.
- Honoring `O_NONBLOCK`/`O_APPEND` (B2.3b, gate-sequenced).
- hostfs inode-faithful dir tracking (needs a host dirent/stat-by-handle ABI —
  separate future slice; permanent degraded mode here, tracked).
- Overlay inode-anchoring MAY be deferred to a follow-up if the
  upper/lower/whiteout/copy-up demux proves too large in-slice — then overlay
  mounts run degraded mode as a tracked matrix exception.
- `O_DIRECTORY`/`AT_*` flag completeness beyond what #59 needs.
- A general inode-tree namespace rewrite of ramfs (name→child-inode trees). We
  re-key paths on rename rather than restructure to a tree; the tree refactor is
  a larger separate effort if ever needed.

## Revision note

The first draft (commit `c7bba40`) under-scoped this as a "VFS-internal
correctness change (no new behavior)" with ramfs at "medium" effort and no
mention of recursive rename or mount boundaries. PR #63 review (verified against
`vfs.rs`) showed (a) ramfs has no directory inodes, (b) recursive subtree rename
is a hard prerequisite and _is_ new behavior, (c) a single-backend component
walk regresses cross-mount `openat`, (d) the default-`None` contract needs a
defined chdir/fchdir/cwd degraded mode, (e) removed-dir create must be `ENOENT`
not "resolve via held inode". All corrected above; scope re-stated as a real VFS
rewrite (matching issue #59's original framing).

A second PR #63 re-review pass (citations re-verified against `vfs.rs`/
`kernel.rs`/`dispatch/fs.rs`) tightened precision before merge: (f) the inode
counter is **per-backend** `next_id` (ramfs `vfs.rs:469`), not the single shared
counter the `vfs.rs:23` doc-comment's wording suggested; (g) `entry_type` cited
at its real sites (`vfs.rs:179`/`712`), not "~L170"; (h) the tar backend named
exactly (`TarLayerBackend`); (i) an explicit dir-inode **isolation invariant**
(no `refcount`/`inodes` entries for dirs) added so the "OFDs unaffected" claim
is grounded; (j) symlink-mid-walk and fork/`clone` cwd-inode tests added (were
described but unenumerated); (k) the `parity-baseline.toml` row uses the quoted
`slice = "B<n>"` form and defers slice assignment to the reviewer per that
file's own process; (l) the `dispatch/fs.rs:197` absolute-path `debug_assert!`
noted as preserved. No scope change — accuracy only.

A third PR #63 re-review pass removed remaining implementation ambiguity: (m)
`resolve_at` no longer requires symlink inodes — symlink filetype is enough to
trigger path-based fallback; (n) inode-anchored `openat` paths containing
`.`/`..` reconstruct and delegate before backend component lookup; (o)
`Process.cwd`'s inode-backed shape is a `PathResolver` invariant for every
relative-path syscall, not only `openat`, `fchdir`, or `getcwd`; (p) refreshed
cwd paths compose `dir_path` with the owning mount's absolute prefix before
returning or caching.
