## Problem

`openat(dirfd, path, flags)` and `fchdir(dirfd)` are not POSIX-faithful: a
directory fd must track the directory **object** (inode), so resolution
stays correct after the directory is renamed or unlinked. Today
`FdEntry::Directory { path: Vec<u8> }` (`kernel.rs:153`) stores only the
path **snapshot** captured at `open()` time; `sys_openat`
(`dispatch/fs.rs:~185`) clones it and joins the child onto it, so after a
rename it resolves the stale path. `fchdir` has the same root cause; a
`FIXME(#59)` marks the site.

```
fd = open("/base", O_DIRECTORY)
rename("/base", "/renamed")
openat(fd, "x", O_CREAT|O_WRITE)   // POSIX: /renamed/x ; actual: /base/x (wrong)
```

`VfsBackend` + `MountTable` are path-keyed for directories; every backend
already allocates stable inode ids and keeps a pathâ†’inode map, but **none
expose inodeâ†’path or component-resolution-from-an-inode**, so the fix
cannot live at the dispatch layer alone â€” it needs a small `VfsBackend`
trait addition.

## Approach (decided: trait dir-inode API)

Add three **default-`None`** methods to `VfsBackend` so existing and
embedder-supplied backends opt in without breaking the pluggable-VFS
contract:

```rust
trait VfsBackend {
    /// Stable inode of `path` iff it is a directory.
    fn dir_inode(&self, path: &[u8]) -> Option<u64> { None }
    /// Resolve ONE component `name` under `dir_inode`.
    /// Returns (child_inode, entry_type) â€” type 3=dir,4=file,â€¦
    fn resolve_at(&self, dir_inode: u64, name: &[u8]) -> Option<(u64, u8)> { None }
    /// Reverse: current absolute (mount-relative) path of a dir inode.
    fn dir_path(&self, dir_inode: u64) -> Option<Vec<u8>> { None }
}
```

`MountTable` gains thin pass-throughs returning `(MountId, inode)` /
`(MountId, inode, type)`, mirroring the existing file-OFD `(mount_id,
inode)` pattern (`kernel.rs:488`).

**Kernel-side:**

- `FdEntry::Directory { path }` â†’ `FdEntry::Directory { mount_id: MountId,
  dir_inode: u64 }`. (One variant; all construction/read sites in
  `dispatch/fs.rs` updated: `sys_open` O_DIRECTORY, `sys_openat`,
  `fchdir`, `chdir`.)
- `sys_openat`: resolve the relative path **component-by-component** via
  `resolve_at`, starting from the fd's `(mount_id, dir_inode)`; absolute
  paths and `AT_FDCWD` keep delegating to `sys_open`. Symlink handling for
  the final target reuses the existing `sys_open` path logic by
  reconstructing the resolved absolute path via `dir_path` at the final
  step (keeps symlink semantics centralized; no symlink-walk duplication
  in the component loop â€” a non-dir/symlink mid-walk falls back exactly as
  `sys_open` would).
- `Process.cwd`: store `(mount_id, dir_inode)` plus a cached path. `chdir`
  resolves to an inode; `fchdir` copies the fd's `(mount_id, dir_inode)`;
  `getcwd` returns `dir_path(inode)` and refreshes the cache, falling back
  to the cached string if the backend returns `None`.

## Testing
