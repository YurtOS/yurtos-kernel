# B1.2 — wait state-model: non-lossy `record_exit` + blocking `EINTR` (issue #99)

Issue #99, the deferred half of **#70** (2026-05-17 holistic review finding
**H5**, cross-validated). Part of the full-parity initiative (tracking #52 /
#71, umbrella #57). **Predecessor: #70 part 1 shipped as PR #98** (`waitid`
signal-death → `CLD_KILLED`, zero-host-coupling). #99 is the explicitly
host-coordinated / process-model remainder #98 deferred — it builds **on** #98,
not over it. **This PR carries the design only**; the implementation lands as
TDD sub-slices after reviewer sign-off on the load-bearing ABI decision below.

## Problem (issue #99)

Two distinct `wait`-family defects survive PR #98, both genuinely
host-coordinated:

### P1 — blocking wait returns `-EAGAIN` (not a valid `wait` errno)

`wait_response`'s no-child path returns `-EAGAIN` on **both** the WNOHANG and
the non-WNOHANG (true-blocking) branches
(`packages/kernel-wasm/src/dispatch/process.rs:1078-1086`), and `waitid`'s
no-waitable-child path returns `-EAGAIN` without WNOHANG
(`process.rs:1206-1210`). `EAGAIN` is **not** a valid `wait` errno — Linux
`wait`/`waitpid`/`waitid` return only `ECHILD`/`EINTR`/`EINVAL`. The audit's
stopgap (until the AsyncBridge blocking path lands — out of scope, still
deferred) is `-EINTR` for the true-blocking path.

The coordination constraint: the deno host's blocking-wait *emulation loop keys
on `-EAGAIN`* —
`while (rc === -EAGAIN && pid > 0 && options?.processEvents)` at
`packages/kernel-host-interface-deno/wasm-kernel-imports.ts:791`. The loop
re-issues `SYS_WAIT` after each `waitForProcessExit` event until the child is
reapable. If the kernel unilaterally returns `-EINTR` on the blocking path, the
loop's guard never matches, `host_wait` returns `-EINTR` on the *first* poll,
and event-driven blocking emulation regresses (every `wait()` for a
not-yet-exited child spuriously fails with EINTR instead of waiting). So the
kernel sentinel and the host loop must change **together**, with a precisely
specified contract (DESIGN DECISION 2 below).

Note `abi/src/yurt_process.c:109-115` already handles **both** `-EAGAIN` (→
return 0, WNOHANG semantics) and `-EINTR` (→ `errno=EINTR`, return -1) from
`yurt_host_wait` — the guest libc is already correct; only the kernel sentinel
+ the deno host loop need the coordinated change.

### P2 — `record_exit` is lossy: exit-code vs signal-death is unrecoverable

`kernel_record_exit(pid, exit_status: i32)` stores `status` verbatim into
`Process.exit_status: Option<i32>`
(`packages/kernel-wasm/src/lib.rs:449-454` →
`process.rs:1026-1046`; field at
`packages/kernel-wasm/src/kernel.rs:379`). The only "discriminator" anywhere is
the `$?`-style `[128, 192)` ⇒ "killed by signal `status-128`" convention that
#98 taught `waitid` to *decode* on the read side
(`process.rs:1221-1231`, the `(128..192).contains(&status)` arm) and that
`abi/src/yurt_process.c:97-101` (`encode_wait_status`) and the deno host
(`wasm-kernel-imports.ts:811-812`) already assume.

This convention is **lossy by construction**:

- A genuine `exit(143)` is byte-identical to `SIGTERM` (15) death — both arrive
  as `status == 143`. `WIFEXITED`/`WIFSIGNALED` cannot disambiguate them.
- `CLD_DUMPED` / `WCOREDUMP` is **unrepresentable**: there is no core-dump bit
  anywhere in the path. #98's own code comment says so explicitly
  (`process.rs:1224-1228` neighborhood: *"CLD_DUMPED is not expressible — the
  8-byte wait record carries no core-dump bit"*).
- The wasmtime host only ever passes the raw `proc_exit` code via
  `UserProcess::last_exit()`
  (`kernel_host_interface.rs:3107-3109` `run_pending_spawns`;
  `:3528-3532` the fork/spawn `child_exit` path; `wasi_shim.rs:233`
  `last_exit = Some(rval)`). There is **no** host-side path that synthesizes
  `128+signo` from a real signal today, so signal-death is not merely lossy —
  it is currently *unproduceable* end-to-end on wasmtime. Any `[128,192)` value
  observed is necessarily a guest that literally `exit()`ed with that code,
  i.e. the ambiguity is real, not theoretical.

The fix: a real exit-vs-signal(+core) discriminator carried through
`record_exit` into `Process`, surfaced unambiguously through both the 8-byte
`sys_wait` record and the 20-byte `sys_waitid` siginfo, and packed by
`yurt_wait_result_v1` so `WIFEXITED`/`WIFSIGNALED`/`WEXITSTATUS`/`WTERMSIG`/
`WCOREDUMP` round-trip, with `waitid` emitting `CLD_DUMPED` for core dumps.

## DESIGN DECISION 1 — the `kernel_record_exit` ABI shape (load-bearing)

This is the decision a reviewer must sign off before plan/TDD. `record_exit`
is a **host-control export** (`#[no_mangle] extern "C"`,
`lib.rs:449`), **not** a method-id-dispatched syscall — it is *not* in
`yurt_abi_methods.toml`, so the wasmtime drift test
(`kernel_host_interface_method_ids_match_yurt_abi_methods_toml`) does **not**
cover it. The wire contract is the raw typed-func tuple, currently
`TypedFunc<(u32, i32), i64>`
(`kernel_host_interface.rs:1947`, bound at `:2584-2585`; JS shape at
`mod.ts:3256-3258`). Any change touches: wasmtime
`get_typed_func` + `record_exit` wrapper (`:2238-2242`, `:2865-2871`) + both
exit-computing callers (`run_pending_spawns:3107-3109`,
fork-spawn `child_exit:3528-3532`); the JS host `recordExit`
(`mod.ts:1559-1564`, `:3544-3546`); the deno registry shims
(`wasm-kernel-imports.ts:400-401`, `:446-447`); and the deno/JS host test
doubles.

### Option (a) — add a third arg: `kernel_record_exit(pid, code, signo_core)`

`TypedFunc<(u32, i32, u32), i64>`. Third arg packs the discriminator:
`0` = normal exit (`code` is the exit code); otherwise low byte = terminating
signo, bit 8 = core-dumped. Kernel constructs
`ExitInfo::Exited(code)` when `signo_core == 0`, else
`ExitInfo::Signaled { signo, core }`.

- **Pro:** explicit, self-documenting, zero ambiguity, trivially testable, no
  sentinel-value cleverness. Mirrors the existing
  `record_thread_exit_authenticated` precedent (a parallel 4-arg host-control
  export already added for an analogous reason —
  `kernel_host_interface.rs:1942`,`:2573`), so the codebase already accepts
  "widen a host-control export's tuple when the old one is lossy".
- **Con:** signature change to a host-control export → wasmtime
  `get_typed_func` tuple change + JS host + two deno shims + test doubles. ~6
  call sites. Bounded and mechanical (the blast radius is enumerated above);
  **not** wire-compat with an unrecompiled old host, but every host in this
  repo is rebuilt from source in lockstep (no external embedders pin this
  export), so that is acceptable here.

### Option (b) — keep `(u32, i32)`, encode a non-lossy sentinel in `i32`

Reuse the single `i32`: e.g. negative magnitude region for signals, a reserved
high bit for core. Concretely `status >= 0` = exit code; `status < 0` =
`-( (signo) | (core ? 0x100 : 0) )`.

- **Pro:** zero signature change — no host/test-double churn at all; the
  smallest possible diff.
- **Con:** re-introduces exactly the implicit-encoding disease #99 exists to
  *kill*. It is "a less-lossy `$?` convention", not a real discriminator: every
  reader must know the codec, the codec is undocumented in any type, and a
  future fourth state (stopped/continued, when WSTOPPED lands) needs yet
  another sentinel carve-out. Trades a one-time mechanical signature change for
  permanent semantic debt — directly against the project's
  *"no JSON / no implicit codecs at the ABI boundary; typed params"* principle.

### Option (c) — a new export `kernel_record_exit_v2`, deprecate the old

Add `kernel_record_exit_v2(pid, code, signo_core)`; keep the old export as a
thin shim (`signo_core = 0`) for any out-of-tree caller.

- **Pro:** strict additive wire-compat; old callers keep working untouched.
- **Con:** there are **no** out-of-tree callers — every host is in this repo
  and rebuilt together. A `_v2` export buys compat we don't need, leaves a dead
  lossy `v1` shim forever, and still requires every *in-tree* host to migrate
  to `_v2` to actually get the fix (same blast radius as (a) **plus** a
  permanent legacy export). Versioned exports earn their keep only with pinned
  external consumers.

### Recommendation: **Option (a)** — widen to `(u32, i32, u32)`

Rationale: (1) it is the only option that makes the discriminator a *typed,
explicit* part of the contract, satisfying the project's no-implicit-codec
principle and leaving room for future states without new sentinels; (2) the
host-coordination cost is real but bounded, fully enumerated, mechanical, and
has direct in-repo precedent (`record_thread_exit_authenticated`); (3) wire
back-compat is a non-constraint here (no pinned external embedders; all hosts
rebuilt in lockstep), which is exactly the condition under which (c)'s
versioning overhead is pure cost; (4) it composes cleanly with #98 — #98's
`waitid` read-side `(128..192)` decode is *replaced* by reading the explicit
`ExitInfo`, and #98's two tests (`waitid_reports_signal_death_as_cld_killed`,
`waitid_normal_exit_stays_cld_exited`) become regression locks that must stay
green through the representation change (the *observable* siginfo for the
existing inputs is unchanged; only the internal carrier and the new
core-dump/true-exit-143 cases are added).

**Reviewer must confirm:** there are no out-of-tree `kernel_record_exit`
consumers (the assumption that makes (a) > (c)). If any pinned external embedder
exists, the decision flips to (c).

## DESIGN DECISION 2 — the deno host-loop coordination contract

The kernel sentinel and `host_wait`'s loop change together. The contract:

**Kernel side (`process.rs`):**

| caller path | no waitable child | sentinel |
|---|---|---|
| `wait_response`, WNOHANG set | yes | `-EAGAIN` (POSIX would-block; unchanged) |
| `wait_response`, blocking (no WNOHANG) | yes | **`-EINTR`** (was `-EAGAIN`) |
| `waitid`, WNOHANG set | yes | success + zeroed siginfo (unchanged, #54) |
| `waitid`, blocking (no WNOHANG) | yes | **`-EINTR`** (was `-EAGAIN`) |
| any, no children at all | — | `-ECHILD` (unchanged) |

`-EAGAIN` survives **only** as the WNOHANG would-block sentinel on
`wait_response`; it is no longer used for the blocking path anywhere.

**Deno host side (`wasm-kernel-imports.ts` `host_wait`, currently `:781-801`):**

The loop currently keys on `-EAGAIN`. New contract: the host distinguishes
*which* sentinel it got by *whether it asked for WNOHANG*.

- The host **always issues the kernel `SYS_WAIT` with WNOHANG forced on**
  (set flags bit 0) so the kernel takes the WNOHANG branch and returns the
  `-EAGAIN` would-block sentinel — the host owns the blocking emulation, the
  kernel must never see a blocking request. The loop then becomes
  `while (rc === -EAGAIN && pid > 0 && options?.processEvents)` — *unchanged
  guard*, because the kernel still returns `-EAGAIN` on the WNOHANG path. After
  the loop, if the **guest** asked for blocking (caller did not pass WNOHANG)
  and the loop exited still `-EAGAIN` only because `processEvents` is absent
  (no event source to wait on), the host returns **`-EINTR`** (a valid
  blocking-wait errno the guest libc already maps —
  `yurt_process.c:112-115`), never `-EAGAIN`, to the guest. If the **guest**
  asked for WNOHANG, the host returns `-EAGAIN` straight through (guest libc
  maps it to `waitpid → 0`, `yurt_process.c:109-111`).

  This is the **recommended** sub-option: it keeps the kernel's blocking-path
  change minimal (the `-EINTR` is still authored — for direct kernel callers
  like the wasmtime `wait_process` host-control path and unit tests — but the
  deno loop never depends on it because it forces WNOHANG). The kernel and host
  stay consistent: kernel's WNOHANG `-EAGAIN` is the *internal* would-block
  signal; the guest never sees a kernel blocking `-EAGAIN` because the deno
  loop never issues a kernel blocking wait, and the wasmtime/JS direct callers
  get the POSIX-valid `-EINTR`.

- Symmetric handling must be added for any future `host_waitid` deno binding.
  **There is none today** (deno binds only `host_wait` / `SYS_WAIT`; no
  `host_waitid`/`SYS_WAITID` binding exists), so the deno loop change is
  confined to the single `host_wait` block; the kernel `waitid` blocking-path
  `-EINTR` is exercised by kernel unit tests and the wasmtime path only.

**Why this exact split:** the kernel must emit a *POSIX-valid* blocking errno
(`-EINTR`) for any direct/blocking kernel caller (wasmtime `wait_process`,
unit tests) because those have no emulation loop; the deno host, which *does*
own an event-driven emulation loop, must not be handed a blocking sentinel it
would treat as a terminal failure, so it forces WNOHANG and keeps owning the
retry. Both sides agree on the invariant: **the guest never observes
`-EAGAIN` from a blocking `wait`, and never observes `-EINTR` spuriously while
an event source could still make progress.**

## Testing (TDD red→green)

Kernel `#[cfg(test)]` via `dispatch()` / the `record_exit` + `kernel_wait`
exports (mirroring #98's `dispatch/tests.rs` `record_exit_req` /
`decode_siginfo` helpers, which must be extended for the new 3rd arg):

P2 — discriminator round-trip:

1. **`record_exit` normal exit → `wait` reports exit-code, not signal.**
   `record_exit(pid, code=42, signo_core=0)` → `sys_wait` 8-byte record
   decodes (via the chosen format) to `WIFEXITED && WEXITSTATUS==42 &&
   !WIFSIGNALED`. Red: today `exit_status` is a bare `i32`.
2. **`record_exit` signal death → `wait`/`waitid` report `WIFSIGNALED` +
   `WTERMSIG`, NOT `WIFEXITED`.** `record_exit(pid, code=0, signo_core=15)` →
   `sys_wait` ⇒ `WIFSIGNALED && WTERMSIG==15`; `waitid` ⇒ `CLD_KILLED` /
   `si_status==15` (extends #98's
   `waitid_reports_signal_death_as_cld_killed`).
3. **The genuine-`exit(143)` disambiguation (the load-bearing P2 case).**
   `record_exit(pid, code=143, signo_core=0)` ⇒ `WIFEXITED &&
   WEXITSTATUS==143 && !WIFSIGNALED`; vs `record_exit(pid, code=0,
   signo_core=15)` ⇒ `WIFSIGNALED && WTERMSIG==15`. Red **today**: both collapse
   to status 143 / both decode as `CLD_KILLED 15` (the exact #99 bug).
4. **Core dump → `waitid` `CLD_DUMPED`, `wait` `WCOREDUMP`.**
   `record_exit(pid, code=0, signo_core = 6 | CORE_BIT)` (SIGABRT+core) ⇒
   `waitid` `si_code == CLD_DUMPED (3)` / `si_status==6`; `sys_wait` record ⇒
   `WIFSIGNALED && WTERMSIG==6 && WCOREDUMP`. Red: `CLD_DUMPED` currently
   unrepresentable.
5. **#98 regression locks stay green:**
   `waitid_normal_exit_stays_cld_exited` (code 42 ⇒ `CLD_EXITED`/42) and
   `waitid_reports_signal_death_as_cld_killed` (now driven by the explicit
   `signo_core=15` instead of the literal `128+15`) must both remain green
   through the representation change — proves #99 builds on #98.
6. **All non-wait `exit_status` readers preserved:** `list_processes` /
   `/proc` snapshot still reports zombie vs running correctly; fork guard
   (`kernel.rs:717`) still refuses to fork an exited parent; kill still skips
   zombies (`kernel.rs:1443`,`:1465`). Existing tests for these must stay
   green through the `Option<i32>` → `Option<ExitInfo>` refactor (refactor
   discipline, mirroring the B2.9 `write_stat_record` extraction guard).

P1 — blocking errno:

7. **Blocking `wait_response` (no WNOHANG), live child not exited → `-EINTR`,
   NOT `-EAGAIN`.** Red: `process.rs:1082-1086` returns `-EAGAIN`.
8. **WNOHANG `wait_response`, no exited child → still `-EAGAIN`** (regression
   lock — the WNOHANG would-block sentinel must NOT change).
9. **Blocking `waitid` (no WNOHANG), no waitable child → `-EINTR`, NOT
   `-EAGAIN`.** Red: `process.rs:1210`.
10. **WNOHANG `waitid`, no waitable child → still success + zeroed siginfo**
    (regression lock, #54 contract unchanged).

Deno parity (`packages/kernel-host-interface-deno/__tests__/
wasm-kernel-imports_test.ts`, extending the existing
`host_wait converts the kernel wait record to yurt_wait_result_v1` /
`host_wait reports kernel signal deaths as wait signals` at `:685`,`:711`):

11. **`host_wait` blocking emulation still retries on event** (the
    `processEvents` loop) — assert the loop still re-issues `SYS_WAIT` after
    `waitForProcessExit` and only the final mapping changed; no spurious
    `-EINTR` while the event source can progress.
12. **`host_wait` true-blocking with no event source → returns `-EINTR`**, not
    `-EAGAIN`, to the (simulated) guest.
13. **`host_wait` WNOHANG passthrough → `-EAGAIN`** reaches the guest (libc
    maps to `waitpid → 0`).
14. **`host_wait` core-dump signal → `yurt_wait_result_v1.flags` carries the
    `WCOREDUMP` bit** and `signal` is the real signo (replacing the
    `:811-812` `>=128` heuristic), `exit_code` 0.

### B0-differ / parity-baseline implications

The TS kernel encodes wait status differently (it does not have the Rust
`Process.exit_status` representation). If a Phase-7 / B0 conformance fixture
exercises signal death, core dump, or blocking-wait errno, the TS-vs-Rust
differ will flag a divergence the moment the Rust side becomes *more* correct.
Per the established discipline (process-model-completion spec, "Tracked
divergences"), **no sub-slice merges with an un-baselined gate divergence**:
either (a) the TS side is brought to the same encoding in the same slice, or
(b) the divergence is explicitly baselined in the differ with a matrix note
("Rust waitstatus is POSIX-faithful; TS uses the legacy `$?` compression —
tracked, Rust is the reference"). The spec does **not** pre-decide which; the
reviewer/plan picks per how the corpus actually exercises wait. Flagged here so
it is measured, not asserted.

## Non-goals

- **Real AsyncBridge blocking suspension.** `-EINTR` is the audit-endorsed
  *stopgap* for the true-blocking path until the AsyncBridge wires actual
  process suspension/resume (its own stack-wide initiative). #99 does **not**
  make blocking `wait` actually block; it makes the non-blocking-stopgap return
  a *POSIX-valid* errno instead of an invalid one, coordinated so deno's
  event-driven emulation is unaffected.
- **WSTOPPED / WCONTINUED** (stop/continue siginfo) — `record_exit` models only
  terminate (exit | signal[+core]); job-control stop/continue is a separate
  process-model slice. The `ExitInfo` enum is shaped so a third
  `Stopped`/`Continued` variant can be added later without another sentinel.
- **Signal-sender permission / `kill` credential gates**, RT-signal queue
  semantics, vfork, pthread_cancel — other B1 slices.
- Anything outside #99's four acceptance boxes.

## Host-ABI blast radius (be honest; gate-sequenced)

This slice is **not** additive like B2.8/B2.9. It changes a host-control export
signature and a wire record. The full surface, gate-sequenced as one
coordinated change (not piecemeal):

1. **kernel-wasm:** `Process.exit_status` type; `record_exit`,
   `wait_response`, `waitid`; `lib.rs` `kernel_record_exit` export tuple;
   `kernel.rs` `list_processes` / `ProcessListEntry` / fork guard / kill
   zombie-skip; `yurt_abi_methods.toml` `sys_wait` + `sys_waitid` docs.
2. **wasmtime host:** `record_exit` `TypedFunc` tuple
   (`kernel_host_interface.rs:1947`,`:2584`), the `record_exit` wrapper
   (`:2238`,`:2865`), and **both** exit-computing callers
   (`run_pending_spawns:3108`, fork-spawn `child_exit:3530`) — these must now
   compute/pass `signo_core` (still `0` until a signal-death producer exists,
   but the call shape changes).
3. **JS host:** `recordExit` (`mod.ts:1559`,`:3544`) + the typed
   `kernel_record_exit` cast (`:3256`).
4. **deno host:** the two `recordExit` registry shims
   (`wasm-kernel-imports.ts:400`,`:446`) and the `host_wait` loop/mapping
   (`:781-822`).
5. **guest libc:** `yurt_process.c` — `encode_wait_status` (`:97-101`) gains
   the `WCOREDUMP` (`0x80`) bit when `result.flags` carries it; `waitpid`
   already handles `-EINTR` (`:112-115`) and `-EAGAIN` (`:109-111`) so the
   blocking-errno half needs **no** `yurt_process.c` change. **No
   `yurt_wait_result_v1` struct change** (`yurt_abi.h:80-85` already has
   `signal` + `flags`).
6. **test doubles:** deno/JS host `recordExit` capturing mocks
   (`wasm-kernel-imports_test.ts:1404`, `kernel-host-interface_test.ts:637`
   etc.) take the new arg.

Because the export signature change must land atomically with the host
callers (a mismatched tuple is a `get_typed_func` instantiation failure, not a
runtime soft-fail), this is **one coordinated PR**, not stacked slices — and it
sequences after the parity gate so the TS-vs-Rust waitstatus divergence
(B0-differ) is measured, not asserted.
