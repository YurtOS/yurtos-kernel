# B2.8 ‚ÄĒ fcntl(F_GETFL) access-mode bits (issue #60)

Issue #60 converted to a PR and folded into the big plan as a tracked
B2 sub-slice. Part of the full-parity initiative (tracking #52, umbrella
#57). Stacks on #55 (B2) ‚ÄĒ it fixes `sys_get_file_status_flags`, which
#55 introduces; base branch = `parity-b2-fd-vfs`.

## Problem (issue #60)

`get_file_status_flags` (F_GETFL) returns only `OpenFileDescription.
status_flags` (the `O_APPEND|O_NONBLOCK` subset). It never includes the
access-mode bits, so for **every** fd `flags & O_ACCMODE == O_RDONLY`.
That is a *wrong answer* ‚ÄĒ musl / CPython `os.get_blocking` / libuv
branch on `(flags & O_ACCMODE)`.

## Scope (B2.8 ‚ÄĒ cargo-unit-testable)

The open ABI carries a single "writable" bit (`flags & 0b001`), not a
2-bit POSIX access mode ‚ÄĒ the libc shim maps both `O_WRONLY` and
`O_RDWR` to `writable=1`, so O_WRONLY-vs-O_RDWR is **unrecoverable at
the kernel** by construction. The achievable, correct fix:

- F_GETFL returns `accmode | status_flags` where `accmode =
  if ofd.writable { O_RDWR (2) } else { O_RDONLY (0) }`.

This fixes the actual bug (no longer always `O_RDONLY`): a read-only fd
reports `O_RDONLY`, a writable fd reports non-`O_RDONLY` ‚ÄĒ exactly what
the affected callers test. No new OFD field is needed; `writable`
already carries the only access information the ABI provides (the
issue's "add an OFD access-mode field" assumed WRONLY/RDWR were
distinguishable; they are not under this ABI).

`O_RDONLY=0 O_WRONLY=1 O_RDWR=2 O_ACCMODE=3` (Linux/musl).

## Residual (tracked, not a kernel bug)

True `O_WRONLY` (write-only, reads ‚Üí EBADF) is unrepresentable until
the open ABI carries the full 2-bit access mode end-to-end (libc shim
+ `yurt_abi` + kernel) ‚ÄĒ a larger ABI change, **not** B2.8. The folded
`pread`/`read` O_WRONLY‚ÜíEBADF item from #60 is therefore also moot
under the current ABI (no pure O_WRONLY fd exists); it re-activates
only if/when the ABI gains a real access mode. Recorded in #60.
